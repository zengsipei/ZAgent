# ADR-0007: 公网接入改道——Tailscale 组网优先、IPv6 直连兜底，应用层完整认证保留为纵深

- 状态：已接受（2026-07-16 owner 签字；初稿与二次改道沿革见下）
- 日期：2026-07-12；2026-07-16 人在环审阅时二次改道（见「决策沿革」）
- 替代：ADR-0003 第 2 条（CF Tunnel + CF Access 一期路线）；其第 1（隧道无关）、3（不可关闭的底线）条延续并在此落地；第 4 条绑定条款对主路径不再触发（Tailscale 恢复了设备级边缘认证），对 IPv6 兜底路径由决策 ④ 预先满足

## 背景

Spike #1 实测否决了 CF Tunnel 路线：各路径击键回显 p50 585–943ms（阈值 200ms），入口腿 + 回程腿双跨太平洋，结构性超标不可优化，终局 no-go。

同期实测（同一物理链路）给出的可用性数据点：

- **Tailscale P2P：p50 = 56ms**——手机蜂窝（移动 2409:）⇄ 电信家宽（240e:）直连未走 DERP，WireGuard 数据面
- IPv6 裸直连：手机蜂窝 → 家宽入站 TCP 直连成功；此前拦截元凶为光猫「安全 → 防火墙 → 防攻击保护」开关
- 两者是**同一条 v6 物理链路**，延迟同级

## 决策沿革

本 ADR 初稿（2026-07-12，状态：提议）定 **IPv6 裸直连**为主路径、VPS 中继（frp）兜底、WebRTC 列后续方向。代码与部署面据此落地并合入（应用层完整认证 + Caddy public profile + `docs/deploy-public.md`）。

2026-07-16 人在环审阅签字环节，owner 翻案改选 **Tailscale 组网**为主路径。依据：

- **体验/延迟对等**：Tailscale P2P 与 IPv6 直连是同一条链路，p50≈56ms，真机已日常用过。
- **安全更优**：光猫「防攻击保护」可以**重新打开**，家中全部设备不再裸对公网 v6 扫描；宿主机零入站端口。WireGuard 设备密钥认证在连接建立前完成——ADR-0003 曾要的「边缘认证」以另一种形式回归。
- **当年弃选的顾虑已消解**：ADR-0003 背景里排除 Tailscale 的唯一理由是「需每台访问设备装客户端」。对单用户自用工具（ADR-0001），访问设备只有 owner 自己的手机与桌面，且**均已安装登录**（spike #1 期间部署，上面那个 P2P 数据点即出自它们）；2026-07-13 人在环测试已在 `tailscale serve` 通路上跑通全链路。
- **运维更轻**：主路径免域名/DNS API/DDNS/证书；IPv6 直连那套的运维前提（光猫开关不被固件升级重置、AAAA 记录跟住动态前缀）降为兜底启用期间才存在。

审阅时翻案正是「提议」状态的用途，流程正当。此改道**不删任何已合代码**：IPv6 直连整套从主路径降级为已就绪兜底（决策 ③）。

## 决策

1. **主路径：Tailscale 组网**——Hub 容器仍只绑 `127.0.0.1:7433`（ADR-0002 端口映射不变，「隧道无关」延续），宿主机跑 `tailscale serve` 在 tailnet 内做 HTTPS 反代（`*.ts.net` 自动证书）。访问设备加入同一 tailnet，WireGuard P2P 直连，打洞失败自动退 DERP 中继（慢但可用）。零入站端口、零公网暴露。
2. **应用层完整认证保留，定位从「唯一防线」改为「纵深防御」**——见决策 ④；这也是本次改道不删认证代码的理由。
3. **兜底：IPv6 直连（已就绪，opt-in）**——初稿落地的 `docker compose --profile public up -d`（Caddy TLS sidecar + DDNS-v6 + 完整认证）整套保留，不启用即零暴露。Tailscale 控制面/DERP 不可达导致失联时，开此 profile 顶上；启用期间才需要关光猫防攻击保护，平时保持开启。比原方案的 frp VPS 兜底更现成、免费、免购置。
4. **应用层完整认证（必须、不可配置关闭）**：
   - 底线延续：长随机根 token + Origin 白名单（ADR-0003 第 3 条原样保留）
   - 会话 token 签发：`POST /auth/session` 用根 token 换发 30 天期 HMAC 会话 token（无状态：校验即重算 MAC，不做簿记；吊销 = 换根 token，全部会话一起失效）。前端只持久化会话 token，根 token 不落 localStorage、不留地址栏。
   - 失败限速：认证失败按来源计数（15 分钟窗口 10 次；X-Forwarded-For 最后一跳分桶，信任前提是 Hub 永不直接暴露——能直连它的只有可信反代与本机）。
   - **为什么 Tailscale 已做设备认证还留这层**：(a) tailnet 内任一设备被攻破即可直达 Hub，token 是内层闸门；(b) 手机丢失时设备的 tailnet 成员身份仍在，token 是第二道锁；(c) 它让 IPv6 兜底「随时可一键启用」始终合法——ADR-0003 第 4 条要求无边缘认证的接入必须先有完整应用层认证，保留此层即预先满足。
5. **移出一期**：
   - **frp / VPS 中继**——兜底职责由已就绪的 IPv6 直连承担，中继不再需要；配方留档 `deploy-public.md` 备查。
   - **WebRTC DataChannel P2P**——原动机「浏览器免装客户端打洞」已消失（访问设备本就在 tailnet），从后续方向移除；Tailscale 若长期不可用再议。

## 后果

- 延迟 p50≈56ms 实测达标，手机蜂窝日用可行；相比初稿方案安全边界更收敛（光猫防护复原、零入站端口）。
- **代价一（结构性）：第三方控制面依赖**。数据面 P2P 建立后不依赖 Tailscale 服务器，但换网（蜂窝⇄WiFi）后端点重协调需控制面或 DERP 可达；两者国内无节点，干扰期可能整体失联——对冲即决策 ③。
- **代价二（体验摩擦）：手机 VPN 独占**。iOS/Android 同时只能激活一个 VPN，Tailscale 与 Clash 类代理互斥，日用按需切换（owner 已接受）。设备密钥默认 180 天过期，应在 admin console 对常用设备关闭 key expiry，免半年一断。
- 主路径安全边界 = tailnet 成员资格（WireGuard）+ serve TLS + Hub 应用层认证；兜底启用时按初稿模型（各设备自身防火墙 + Caddy 443 唯一入站口 + TLS + 应用层认证），`deploy-public.md` 有坦白。
- 部署配方：主路径固化于 `docs/deploy.md`「远程接入」节（含 Docker 场景），兜底保留于 `docs/deploy-public.md`。
