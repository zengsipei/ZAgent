# ADR-0007: 公网接入改道——IPv6 直连优先、VPS 中继兜底，应用层完整认证转必须

- 状态：提议（人在环确认后置为已接受）
- 日期：2026-07-12
- 替代：ADR-0003 第 2 条（CF Tunnel + CF Access 一期路线）；其第 1（隧道无关）、3（不可关闭的底线）、4（绑定条款）条延续并在此落地

## 背景

Spike #1 实测否决了 CF Tunnel 路线：各路径击键回显 p50 585–943ms（阈值 200ms），入口腿 + 回程腿双跨太平洋，结构性超标不可优化，终局 no-go。ADR-0003 第 4 条绑定条款由此触发：改用无边缘认证的接入方式时，完整应用层认证从可选转为**必须**，且为切换的前置任务。

同期实测（同一物理链路）：

- Tailscale P2P：p50 = 56ms——可用性锚点，证明链路本身足够快
- IPv6 裸直连：手机蜂窝（移动 2409:）→ 家宽（电信 240e:）入站 TCP 直连成功；此前拦截元凶为光猫「安全 → 防火墙 → 防攻击保护」开关，已定位并关闭

## 决策

1. **主路径：IPv6 直连**——DDNS-v6 + WSS（Caddy sidecar 做 TLS 终结，唯一入站口）+ 应用层完整认证。最轻零依赖。前提：光猫防攻击开关保持关闭；蜂窝侧存在 IPv6。
2. **兜底：国内 VPS 中继**（frp；与将来 WebRTC 的 TURN 合流为同一台 VPS 的两种用法）——蜂窝 v6 缺失或直连不可用时走此路径。
3. **WebRTC DataChannel P2P 列为后续方向**：浏览器端唯一打洞原语，信令可复用 Hub WS，预期延迟与 Tailscale 同级。本 ADR 只定方向，工程量大，另立票落地；一期兜底由 VPS 中继承担。
4. **应用层完整认证（必须、不可配置关闭）**：
   - 底线延续：长随机根 token + Origin 白名单（ADR-0003 第 3 条原样保留）
   - **会话 token 签发**：`POST /auth/session` 用根 token 换发 30 天期 HMAC 会话 token（无状态：校验即重算 MAC，不做簿记；吊销 = 换根 token）。前端只持久化会话 token，根 token 不落 localStorage、不留地址栏。
   - **失败限速**：认证失败按来源 IP 计数（15 分钟窗口 10 次），打满后该 IP 的认证请求一律拒绝到窗口结束。经中继接入时全来源同 IP、退化为全局阈值——防无人值守暴力枚举的兜底目的仍成立。
5. **安全模型变化坦白**：原方案「本地零入站端口」卖点在 IPv6 直连下不再成立。新安全边界 = 光猫防火墙关闭后各内网设备自身防火墙 + 唯一入站口（Caddy 443）+ TLS + Hub 应用层完整认证。Hub 自身仍不监听公网，只被容器内网访问。

## 后果

- 延迟回到 Tailscale 同级（p50 ≈ 56ms），手机蜂窝日用可行——本项目的核心可用性指标达成
- 需要一个域名与 DNS API（DDNS + ACME 证书）；部署步骤固化在 `docs/deploy-public.md`
- 新增运维面：DDNS 稳定性、证书续期、光猫开关被固件升级重置的风险——公网上线实测后再决定是否固化为票（见 map #12）
- 蜂窝 v6 少数场景缺失是已知残余风险：一期由 VPS 中继兜底，中期由 WebRTC P2P 补位
