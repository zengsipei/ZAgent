# ADR-0003: 隧道无关设计，CF Access 边缘认证 + 不可关闭的应用层底线

- 状态：已接受；第 2 条（CF Tunnel + CF Access 路线）被 ADR-0007 替代（spike #1 实测延迟 no-go；现行：Tailscale 主路径 + IPv6 直连兜底），第 4 条绑定条款之完整认证已由 ADR-0007 预先落地（对无边缘认证的 IPv6 兜底路径为唯一防线）；背景中弃选 Tailscale 的理由被 ADR-0007 决策沿革推翻
- 日期：2026-07-09

## 背景

本地机在 NAT 后。可选路径：Tailscale 组网（需每台访问设备装客户端）、自建云中继（工作量翻倍）、现成隧道（CF Tunnel / frp）。一个能执行任意命令的 PTY 暴露到公网入口是本项目最大安全风险；「先跑通、认证后补」不是选项。

## 决策

1. **隧道无关**：Hub 只监听容器内网/localhost，不感知隧道层；cloudflared、frp 均可作为 sidecar 接入。
2. **一期采用 Cloudflare Tunnel + Cloudflare Access**：cloudflared 出站连接（本地零入站端口），Access 在边缘完成身份认证，未认证流量不触达 Hub。需要一个托管在 CF 的域名。
3. **应用层底线写死在 Hub 内、不可配置关闭**：WS upgrade 时校验长随机 token + Origin 白名单（约 20 行）。
4. **绑定条款**：「Hub 无完整应用层认证」与「CF Access 在前」绑定生效。任何时候换成无边缘认证的隧道（如 frp），完整应用层认证（token 签发、失败限速）从可选转为**必须**，且为切换的前置任务。

## 后果

- 安全大头白嫖给 CF，本地代码只承担底线防御（防「换隧道忘前提」与「恶意网页借浏览器发起 WS」）。
- **待验证假设：CF 国内击键延迟可接受**（150–300ms 量级风险）。动工第一周用 ttyd + CF Tunnel 实测；不达标降级 frp + 国内 VPS，同时触发第 4 条条款。
- 终端为打字回显场景，延迟直接决定日用体验——此假设不成立时网络层决策整体重开。
